# Описание компонентов архитектуры

Версия: 3.0  
Дата: 2026-04-21

Данный документ содержит краткое описание всех сервисов, слоёв и компонентов, изображённых на архитектурной диаграмме C4 (Container Diagram).

---

## 🌐 External Systems (Внешние системы)

| Компонент | Роль и назначение |
| :--- | :--- |
| **CRM / UI** | Единое рабочее место трейдера, диспетчера и аналитика. Веб-интерфейс, через который сотрудники:<br>• Видят графики выработки в реальном времени,<br>• Получают уведомления о статусах команд и авариях,<br>• Отправляют команды управления мощностью,<br>• Проверяют и подтверждают торговые заявки перед отправкой на биржу. |
| **API Gateway** | «Входная дверь» платформы для внешнего мира. Выполняет:<br>• Приём запросов от UI,<br>• Аутентификацию и проверку прав доступа,<br>• Маршрутизацию запросов к внутренним сервисам (Command Tracker, Dispatch Service),<br>• Трансформацию внешних команд во внутренние события шины данных (Kafka). |

## 🧩 Platform Services (Бизнес-сервисы платформы)

| Сервис | Основная функция | Ключевая бизнес-ценность |
| :--- | :--- | :--- |
| **Telemetry Service** | Приём сырых данных телеметрии из топика `telemetry.raw`, их валидация, фильтрация шумов и агрегация (например, усреднение с 10 Гц до 1 секунды). Результат публикуется в `telemetry.processed`. | Обеспечивает **чистоту и достоверность** исходных данных для всех downstream-сервисов (прогноз, трейдинг, отчёты). |
| **Forecast Service** | На основе очищенной телеметрии, исторических данных из `TimescaleDB` и внешних погодных факторов строит **почасовой прогноз генерации** для каждой СЭС на следующие сутки. Результат — в `forecast.updated`. | Критически важен для точной подачи заявок на рынок «на сутки вперёд». Ошибка прогноза = финансовые потери на небалансах. |
| **Trading Engine** | «Мозг» торговой стратегии. Использует прогнозы (`forecast.updated`), текущие обязательства по рынку и команды диспетчера для:<br>• Формирования оптимальных заявок на биржу,<br>• Расчёта доступной мощности в реальном времени,<br>• Реагирования на исполнение/сбой команд. | Автоматизирует процесс **максимизации прибыли** на спотовом и балансирующем рынках, снижая влияние человеческого фактора. |
| **Dispatch Service** | Сервис диспетчеризации команд управления мощностью. Принимает решения от трейдера (через `Bid Created Topic` или напрямую) и помещает их в `command.requested`. Также **отслеживает обратную связь** (`executed`/`failed`) для контроля исполнения. | Гарантирует, что **единственный источник истины** о требуемой мощности станции корректно доставлен на все площадки и исполнен. |
| **Notification Service** | Подписан на ключевые события (`bid.created`, `command.executed`, `command.failed`) и доставляет уведомления заинтересованным сторонам (трейдер в UI, e-mail, мессенджеры). | Обеспечивает **мгновенное информирование** о критических изменениях: «Заявка подана», «Команда выполнена», «Сбой на СЭС №2». |
| **Command Tracker** | **Read Model** для отслеживания жизненного цикла каждой команды управления. Слушает все топики с префиксом `command.` и строит в `Redis` проекцию текущего состояния: `PENDING → SENT → EXECUTED/FAILED`. Предоставляет быстрый `GET /status/{id}` через API Gateway. | Даёт трейдеру **полную прозрачность**: в любой момент можно узнать, на какой стадии находится его распоряжение, без необходимости звонить на станцию. |

## 💾 Data Storage (Хранилища данных)

| Хранилище | Назначение |
| :--- | :--- |
| **TimescaleDB** | Специализированная time-series база для хранения агрегированной телеметрии и прогнозов. Оптимизирована для быстрых запросов по временным диапазонам (графики за час, день, месяц). |
| **Redis** | In-memory кэш для хранения оперативного состояния команд и сессий. Обеспечивает минимальную задержку при отображении статуса выполнения в UI. |

## ☀️ Edge Layer (Периферия на каждой СЭС)

| Компонент | Роль |
| :--- | :--- |
| **SCADA / Inverters** | Полевое оборудование: контроллеры инверторов, счётчики, датчики. Источник первичных данных (вольты, амперы, МВт). |
| **Edge Adapter** | Промышленный шлюз, который «говорит» на протоколах конкретной SCADA (Modbus, OPC UA) и транслирует данные в универсальный формат платформы. Также принимает команды из `command.requested` и передаёт их на инверторы. |
| **Local Buffer** | Встроенное в адаптер локальное хранилище (файл / RocksDB). Накапливает телеметрию при обрыве связи с центром и автоматически досылает её при восстановлении. |

## 🔄 Apache Kafka (Event Backbone)

Центральная шина событий, обеспечивающая надёжную асинхронную связь между всеми компонентами. Основные топики:

| Топик | Назначение |
| :--- | :--- |
| `telemetry.raw` | Сырые данные с площадок СЭС. |
| `telemetry.processed` | Очищенные и агрегированные данные телеметрии. |
| `forecast.updated` | Почасовой прогноз генерации для каждой станции. |
| `bid.created` | Сформированная торговая заявка. |
| `command.requested` | Запрос на изменение мощности (единый для всех СЭС, маршрутизация по `plant_id`). |
| `command.sent` | Команда физически отправлена на контроллер инвертора. |
| `command.executed` | Команда успешно выполнена оборудованием. |
| `command.failed` | Ошибка при выполнении команды (таймаут, отказ оборудования). |

---

Эта структура чётко разделяет зоны ответственности и позволяет каждой команде (разработчики Edge, платформы, аналитики) работать независимо, опираясь на контракты топиков Kafka.
